# 제품 설계 초안

## 1. 문제 정의

초등학교 시간표 배정은 단순한 교시 배치가 아니라 아래 자원을 동시에 만족해야 한다.

- 학급별 주간 수업 시수
- 전담교사 가능 시간
- 특별실 사용 가능 시간
- 학년/학급별 고정 일정
- 교사 중복 불가
- 특별실 중복 사용 불가
- 같은 과목의 분산 배치 또는 연속 배치 조건
- 학교별 예외 규칙

이 시스템은 위 조건을 설정 가능한 규칙으로 관리하고, 자동 배정과 수동 조정을 함께 지원해야 한다.

## 2. 목표

### 핵심 목표

- 특별실 사용 시간표와 전담수업 개별 시간표를 한 시스템에서 관리
- 학교마다 다른 조건을 설정값으로 처리
- 자동 배정 후 사람이 쉽게 수정 가능
- 배정 충돌을 즉시 검출
- 결과를 주간 시간표, 교사별 시간표, 특별실별 시간표로 동시에 조회

### 비핵심 목표

- 처음부터 모든 학교 규칙을 완전 자동화하지는 않음
- 첫 버전에서 AI 추천보다 규칙 기반 배정을 우선

## 3. 사용자 유형

- 관리자: 학교 전체 기준정보, 규칙, 배정 실행 관리
- 담임교사 또는 시간표 담당자: 학급 시간표 조회 및 수정
- 전담교사: 개인 시간표 조회
- 특별실 담당자: 특별실 사용 현황 조회

## 4. 기능 요구사항

### 4.1 기준정보 관리

- 학년도, 학기, 주간 교시 수 설정
- 학년, 학급 생성
- 교과목 관리
- 전담교사 정보 관리
- 특별실 정보 관리
- 요일/교시 체계 설정
- 학교 공통 비가용 시간 설정

### 4.2 배정 대상 데이터 관리

- 학급별 과목 주간 시수 입력
- 과목별 담당교사 연결
- 과목별 사용 특별실 연결
- 교사별 가능/불가 시간 입력
- 특별실별 가능/불가 시간 입력
- 학급별 고정 수업 입력
- 학급별 피해야 할 시간 입력

### 4.3 규칙 관리

- 하드 제약 조건
  - 같은 시간에 한 교사는 한 수업만 가능
  - 같은 시간에 한 특별실은 한 수업만 가능
  - 한 학급은 같은 시간에 한 수업만 가능
  - 교사 불가 시간에는 배정 불가
  - 특별실 불가 시간에는 배정 불가
- 소프트 제약 조건
  - 특정 과목은 오전 배치 선호
  - 동일 과목 연속 배치 최소화
  - 체육/음악/미술 등 이동 수업 간 균형
  - 특정 요일 쏠림 방지
  - 전담교사 공강 최소화
- 학교별 사용자 정의 규칙 확장 가능

### 4.4 자동 배정

- 전체 자동 배정
- 특정 학년만 부분 자동 배정
- 특정 교사/특별실 중심 재배정
- 충돌 발생 시 사유 표시
- 여러 후보안 생성 가능

### 4.5 수동 편집

- 드래그앤드롭 또는 셀 선택 기반 편집
- 수동 이동 시 충돌 즉시 검사
- 강제 배정 가능 여부 설정
- 잠금 기능
  - 확정된 수업은 재배정 제외
- 변경 이력 기록

### 4.6 조회 및 출력

- 학급별 주간 시간표
- 교사별 개별 시간표
- 특별실별 사용 시간표
- 충돌 목록
- 미배정 목록
- 인쇄용 출력
- 엑셀 업로드/다운로드

## 5. MVP 범위

첫 버전에서는 아래 범위에 집중하는 것이 현실적이다.

- 학교 1개의 학기 시간표 관리
- 학급, 교사, 과목, 특별실 기본정보 관리
- 학급별 시수 입력
- 교사/특별실 비가용 시간 설정
- 하드 제약 중심 자동 배정
- 학급별/교사별/특별실별 시간표 조회
- 수동 수정 및 충돌 검사
- 엑셀 내보내기

첫 버전에서 제외해도 되는 항목:

- 다중 학교 SaaS 운영
- 고급 권한 체계
- 모바일 앱
- AI 기반 추천
- 복잡한 소프트 제약 최적화

## 6. 배정 엔진 설계 방향

### 6.1 핵심 개념

시간표 배정은 조합 최적화 문제이므로 CRUD와 분리된 엔진 계층이 필요하다.

- 입력
  - 시간 슬롯
  - 학급별 필요 수업
  - 교사 자원
  - 특별실 자원
  - 제약 조건
- 출력
  - 배정 결과
  - 충돌 정보
  - 미배정 정보
  - 규칙 위반 점수

### 6.2 엔진 처리 단계

1. 입력 데이터 정규화
2. 하드 제약 검증
3. 초기 배정 생성
4. 충돌 없는 해 탐색
5. 소프트 제약 점수 계산
6. 결과 저장

### 6.3 추천 구현 방식

초기 구현은 아래 중 하나가 적합하다.

- 방식 A: 백트래킹 + 휴리스틱
  - 구현이 단순하고 제어가 쉬움
  - 작은 규모 학교부터 적용하기 좋음
- 방식 B: CP-SAT 같은 제약 충족 솔버 사용
  - 복잡한 제약 표현에 강함
  - 향후 확장성이 좋음

권장: 초기부터 엔진 인터페이스는 추상화하고, 첫 구현은 휴리스틱 또는 CP-SAT 중 하나로 시작한다.

## 7. 시스템 구조 제안

## 7.1 프론트엔드

- 관리자 설정 화면
- 기준정보 관리 화면
- 시간표 편집 화면
- 충돌/미배정 검토 화면
- 출력 화면

권장 기술:

- Next.js
- TypeScript
- Tailwind CSS 또는 컴포넌트 시스템
- TanStack Table/Query 계열 도입 검토

## 7.2 백엔드

- REST API 또는 tRPC
- 인증/권한
- 기준정보 CRUD
- 배정 실행 API
- 충돌 검사 API
- 출력 데이터 생성 API

권장 기술:

- Next.js 서버 액션 또는 별도 Node.js API
- PostgreSQL
- Prisma

## 7.3 배정 엔진

- 애플리케이션 계층과 분리된 도메인 서비스
- 입력 DTO -> 엔진 모델 변환
- 엔진 실행 결과 -> 저장용 모델 변환
- 향후 별도 워커 프로세스로 분리 가능

## 7.4 권장 프로젝트 구조

```text
src/
  app/
    (dashboard)/
    api/
  components/
    timetable/
    forms/
    ui/
  features/
    classes/
    teachers/
    rooms/
    subjects/
    scheduling/
  lib/
    db/
    auth/
    excel/
  domains/
    timetable/
      entities/
      rules/
      services/
  engine/
    core/
    constraints/
    heuristics/
    adapters/
  types/
```

의도는 다음과 같다.

- `features`: 화면과 API에 가까운 기능 단위 코드
- `domains`: 시간표 도메인 규칙과 서비스
- `engine`: 자동 배정 계산 전용 계층
- `lib`: 공통 인프라 코드

## 8. 데이터 모델 초안

### 주요 엔터티

- SchoolTerm
- Grade
- ClassRoomGroup
- Subject
- Teacher
- SpecialRoom
- TimeSlot
- ClassSubjectRequirement
- TeacherAvailability
- RoomAvailability
- FixedSchedule
- ScheduleEntry
- ConstraintRule
- SchedulingRun

### 관계 예시

- 한 과목은 여러 학급 요구사항과 연결될 수 있음
- 한 과목은 한 명 이상의 담당교사와 연결될 수 있음
- 한 수업 배정은 학급, 과목, 교사, 시간슬롯을 가짐
- 특별실 필요 과목인 경우 특별실도 함께 연결

## 11. 먼저 결정해야 할 사항

아래는 실제 설계 전에 꼭 확정해야 한다.

- 시간표 단위가 교시 고정인지, 블록 수업까지 허용하는지
- 전담수업이 학급마다 동일 패턴인지, 학급별 예외가 많은지
- 특별실이 필수인 과목과 선택인 과목을 구분할지
- 자동 배정 목표가 완전 자동인지, 반자동 보조인지
- 교사 권한 분리 수준이 필요한지
- 엑셀 업로드가 필수인지

## 12. 추천 방향

가장 현실적인 출발점은 다음과 같다.

- 웹 기반 단일 학교용 관리 시스템으로 시작
- 관리자 중심 사용 구조
- 하드 제약 우선 배정 엔진 구현
- 자동 배정 후 수동 보정 가능한 편집기 제공
- 이후 학교별 예외 규칙을 추가하는 방식으로 확장
